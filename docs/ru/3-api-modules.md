# API Modules

## Цель документа

Этот документ описывает **основные API-модули NexusFlow** на уровне backend-дизайна.

Его задача — зафиксировать:

- границы модулей,
- ответственность каждого домена,
- примерную структуру API.

---

# 1. Auth Module

## Назначение
Модуль аутентификации отвечает за identity и access lifecycle пользователя.

## Ответственность за
- регистрация
- вход
- refresh токены
- logout
- смена пароля
- валидация доступа

## Пример эндпоинтов

### `POST /auth/register`
Создание нового аккаунта.

### `POST /auth/login`
Аутентификация пользователя.

### `POST /auth/refresh`
Обновление access token.

### `POST /auth/logout`
Завершение сессии.

### `POST /auth/change-password`
Смена пароля.

## Главные сущности
- User
- RefreshToken / Session

# 2. Users Module

## Ответсвенность за

### `PATCH /users/me`
Обновление собственного профиля.

### `GET /users/{user_id}`
Просмотр публично доступного профиля пользователя.

## Главные сущности

# Access Model

## Team-Scoped Access
Почти все основные модули NexusFlow должны работать через **team-scoped access model**.

Это означает:

- пользователь видит только те команды, где он состоит,
- пользователь видит только те задачи, чаты и summary, которые относятся к его команде,
- доступ должен проверяться на уровне membership.

Это одна из ключевых архитектурных основ продукта.

# API Design Principles

## 1. Resource-oriented API
API должно быть построено вокруг доменных сущностей, а не случайных действий.

## 2. Clear ownership
Каждый ресурс должен иметь понятную область принадлежности:

- user-scoped
- team-scoped
- channel-scoped
- task-scoped

## 3. Predictable naming
Имена endpoint'ов должны быть предсказуемыми и консистентными.

## 4. Thin routers, strong service layer
Логика не должна жить в route handlers.  
Основная бизнес-логика должна быть в service layer.

## 5. Async-friendly architecture
AI и real-time сценарии должны проектироваться с учётом асинхронных процессов.
