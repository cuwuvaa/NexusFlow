# MVP Scope

## Цель документа

Этот документ фиксирует **границы MVP** для NexusFlow.

Его задача — защитить проект от фичекрип и сохранить фокус на первом рабочем релизе.  


---

## MVP Goal

MVP NexusFlow должен доказать, что продукт способен решить базовую проблему:

> **дать технической команде единое пространство для общения, распределения ролей, задач и AI-поддержки контекста.**

Иными словами, первая версия должна позволять команде:

- зарегистрироваться,
- создать команду,
- распределить роли,
- общаться,
- создавать и отслеживать задачи,
- использовать AI для подитоживания коммуникации.

Если это работает стабильно и целостно — MVP успешен.

---

## IN (входит в MVP)

### 1. Authentication
Базовая аутентификация и управление доступом.

Включает:

- регистрация пользователя,
- вход в систему,
- refresh/access tokens,
- logout,
- смена пароля,
- базовая защита маршрутов.

---

### 2. Users
Базовое управление пользователями.

Включает:

- профиль пользователя,
- имя / username / email,
- avatar URL (опционально),
- просмотр собственного профиля,
- обновление базовой информации,

---

### 3. Teams
Работа с командами как основной единицей продукта.

Включает:

- создание команды,
- просмотр команды,
- приглашение участников,
- присоединение участников,
- список участников,
- базовые настройки команды.

---

### 4. Roles
Ролевая модель внутри команды.

Включает:

- системные роли (например: owner, manager, backend, frontend, devops, tester, ai),
- назначение роли участнику,
- отображение роли внутри команды,
- использование ролей для базовой организационной структуры.

> В MVP роли нужны прежде всего для организации команды, а не для сложной системы разрешений.

---

### 5. Tasks
Базовый task management.

Включает:

- создание задачи,
- редактирование задачи,
- назначение исполнителя,
- статусы задач,
- приоритет,
- дедлайн (опционально),
- просмотр списка задач,
- просмотр задач по участнику / по команде.

Это один из центральных блоков MVP.

---

### 6. Chat
Командная коммуникация в реальном времени.

Включает:

- командные чаты,
- отправка сообщений,
- история сообщений,
- базовая real-time доставка,
- привязка сообщений к команде / каналу.

> В MVP достаточно текстового чата без сложных мультимедийных сценариев.

---

### 7. AI Summaries
Первый AI-модуль, встроенный в основной рабочий сценарий.

Включает:

- выжимка сообщений чата,
- выделение ключевых решений,
- краткие итоги обсуждения,
- генерация дайджеста по диалогу / каналу.

Это должен быть **не декоративный AI**, а реально полезная функция снижения коммуникационной перегрузки.

---

## OUT of Scope (не входит в MVP)

Ниже перечислены функции, которые **осознанно исключены** из первой версии.

Они могут быть реализованы позже, но не должны блокировать выпуск MVP.

---

### 1. Calls / Voice / Video
Не входит в MVP:

- voice calls,
- video calls,
- групповые звонки,
- live meeting rooms.

Причина: высокая сложность real-time media и низкая критичность для первой версии.

---

### 2. Board / Shared Canvas
Не входит в MVP:

- визуальные доски,
- whiteboard,
- collaborative canvas,
- Miro-like взаимодействие.

Причина: отдельный крупный продуктовый блок, который сильно расширяет scope.

---

### 3. Architecture Graph Editor
Не входит в MVP:

- визуальный редактор архитектуры,
- drag-and-drop схема сервисов,
- граф зависимостей,
- интерактивная архитектурная карта.

Причина: очень сильная идея, но это уже отдельный крупный модуль, который лучше строить после подтверждения основного продукта.

---

### 4. GitHub Integrations
Не входит в MVP:

- синхронизация репозиториев,
- pull requests,
- issues,
- webhooks из GitHub,
- коммит-активность.

Причина: интеграции резко повышают сложность и должны идти только после стабилизации core платформы.

---

### 5. Advanced Notifications
Не входит в MVP:

- сложные notification rules,
- email notifications,
- push notifications,
- умные reminder pipelines,
- notification preferences matrix.

В MVP допустимы только **простые in-app уведомления**, если они нужны для базового UX.

---

## Non-Goals of MVP

MVP **не пытается**:

- заменить все инструменты команды,
- стать полноценным Slack + Jira + Miro + GitHub в одной версии,
- покрыть все сценарии управления разработкой,
- решить все проблемы engineering collaboration сразу.

MVP должен доказать только одно:

> **команде действительно удобно работать в NexusFlow как в единой операционной среде.**

---

## MVP Success Criteria

MVP можно считать успешным, если команда может пройти следующий сценарий:

1. Пользователь регистрируется
2. Создаёт команду
3. Приглашает участников
4. Назначает роли
5. Создаёт задачи
6. Обсуждает работу в чате
7. Использует AI для получения кратких итогов обсуждений

Если этот сценарий работает:

- стабильно,
- логично,
- без архитектурного хаоса,
- с хорошей backend-основой,

то MVP достиг своей цели.

---

## Scope Control Rule

Любая новая идея, которая возникает во время разработки, должна проходить через правило:

> Если без неё MVP всё ещё решает основную проблему — значит, она не обязательна для MVP.

Это правило должно соблюдаться жёстко.
